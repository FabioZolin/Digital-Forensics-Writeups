# Writeup: Memory Analysis - Ransomware (Blue Team Labs Online)

**Platform:** Blue Team Labs Online (BTLO)  
**Category:** Digital Forensics / Memory Analysis  
**Tools Used:** Volatility 2 (`vol2` / `vol.py`)  
**OS Profile:** `Win7SP1x86`  

---

## Introduction
In this challenge, we are provided with a memory dump (`infected.vmem`) of a Windows 7 machine suspected of being infected by ransomware. The goal of the investigation is to identify malicious processes, reconstruct the execution chain (process tree), discover the infection's point of origin, and recover cryptographic artifacts.

## Phase 1: Initial Triage and Threat Identification
To begin the analysis, the first logical step is to get an overview of the active processes at the time of the dump. By using the `psscan` plugin, we can also identify hidden or terminated processes, even if there was some tampering of the EPROCESS structures.

> vol.py -f infected.vmem --profile=Win7SP1x86 psscan

Scrolling through the process list, an anomalous name immediately stands out. Unlike normal system executables, this one lacks the `.exe` extension and has a name closely related to encryption/decryption operations.

> **Q: Run A process scan to see which processes were running. What is the name of the suspicious process?** > **A:** `@WanaDecryptor`

## Phase 2: Process Tree Reconstruction
Once the payload (`@WanaDecryptor`) is identified, it's crucial to understand what launched it. Analyzing the output of the previous `psscan` command, I checked the **PPID** (Parent Process ID) associated with `@WanaDecryptor`. 

> **Q: What is the parent process ID for the suspicious process?** > **A:** `2732`

Knowing that the parent PID is `2732`, I searched for the process corresponding to this identifier. The result revealed the name of the original executable from which the compromission started.

> **Q: What is the initial malicious executable that created this process?** > **A:** `or4qtckT.exe`

## Phase 3: Deep Dive into Child Processes
The main process (`or4qtckT.exe` - PID 2732) might have spawned other processes besides `@WanaDecryptor`. To verify this, I filtered the `psscan` output, isolating only the activities related to this its PID:

> vol.py -f infected.vmem --profile=Win7SP1x86 psscan | grep "2732"

The output revealed that, in addition to two instances of `@WanaDecryptor`, the malware launched another executable: `taskdl.exe`.

> **Q: If you drill down on the suspicious PID (vol.py -f infected.vmem --profile=Win7SP1x86 psscan | grep (PIDhere)), find the process used to delete files** > **A:** `taskdl.exe`

## Phase 4: Execution Point and OSINT
To understand where the attacker (or the unaware user) launched the initial executable from, I extracted the command-line arguments associated with PID 2732. 

> vol.py -f infected.vmem --profile=Win7SP1x86 cmdline -p 2732

This command revealed the full file path on the disk from which the malware was started.

> **Q: Find the path where the malicious file was first executed** > **A:** `C:\Users\hacker\Desktop\or4qtckT.exe`

Correlating the extracted artifacts—specifically the filenames `@WanaDecryptor` and `taskdl.exe`—a quick OSINT search unequivocally identifies the threat as **WannaCry**, one of the most notorious and devastating ransomware strains in history. Further research into the specific roles of these processes reveals their distinct functions in the attack lifecycle: `@WanaDecryptor` serves as the graphical user interface (GUI) displaying the ransom note, while `taskdl.exe` is a destructive component specifically designed to eradicate system backups and shadow copies, eliminating any chance of data recovery.

> **Q: Can you identify what ransomware it is? (Do your research!)** > **A:** `Wannacry`

## Phase 5: Extraction of Cryptographic Artifacts
The last step requires finding the file containing the public key used by the ransomware. This file is typically generated and opened by the malicious process. I used the `handles` plugin to inspect all "objects" (files, registry keys, etc.) opened by the parent process (PID 2732) and filtered the search for the extension indicated in the challenge (`.eky`).

> vol.py -f infected.vmem --profile=Win7SP1x86 handles -p 2732 | grep ".eky"

The search was successful, revealing the exact name of the file generated by the ransomware to manage the encryption keys.
If it wasn't the search could have been extended to all .eky handles and manually seen if the list wasnt too numerous.

> **Q: What is the filename for the file with the ransomware public key that was used to encrypt the private key? (.eky extension)** > **A:** `00000000.eky`

## Conclusions and Lessons Learned
This investigation confirmed the system compromise by the **WannaCry** ransomware. Based on the behavior of this specific malware family, we can draw the following defensive considerations (Blue Team):

1. **Exploited Vulnerability:** WannaCry is known to spread laterally by exploiting the *EternalBlue* vulnerability (CVE-2017-0144) in the Windows SMBv1 protocol.
2. **Mitigation:** It is crucial to disable the legacy SMBv1 protocol on all network machines and ensure that the **MS17-010** security patch is applied.
3. **Prevention:** The original infection started from the user's Desktop, suggesting a possible compromise via phishing or social engineering. The adoption of an updated EDR (Endpoint Detection and Response) system and network segmentation are essential to block the execution of suspicious payloads and limit their spread.
